# DB設計
## 第一章　データベースの代表的なモデル
### リレーショナルデータベース(RDB)
関係データベースともよばれ、現在最も広く利用されているデータベース。<br>
普通、IT業界で「データベース」といえば、リレーショナルデータベースを指す。<br>
特徴としては、データを人間が理解しやすい二次元表の形式で管理。データの取り扱いが他のデータベースに比べると簡単。
### オブジェクト指向データベース
プログラミング言語の中では、JavaやC++など、オブジェクト指向言語と呼ばれるタイプの言語があるが、データと操作をまとめて「オブジェクト」という単位で管理されるのでそう呼ばれている。<br>
※補足<br>
オブジェクト指向（Object-Oriented Programming：OOP）とは<br>
プログラムを「モノ（オブジェクト）」の集まりとして考えて作る考え方<br>
・性質（データ）・ふるまい（処理）をひとかたまりにて管理することが特徴<br>
例（javaの場合）
```markdown
class Dog {
  String name;
  int age;

  void bark() {
    System.out.println("ワン！");
  }

  void run() {
    System.out.println("走ります！");
  }
}
```
このDogがオブジェクトとなる


### XMLデータベース
web上でやり取りされるデータの形式にXML(HTMLのようにタグでデータ管理を行う言語)を利用とする動きがあった。<br>
特徴としては、リレーショナルデータベースが苦手とする階層構造のデータの扱いを得意とする。<br>
```markdown
<address>
 <name>山田　太郎<name>
 <prefecture>東京都</prefecture>
 :
 :
 </address>
 ```

 ### キー・バリューストア(Key-Value Store:KVS)
 データをKey(識別キー)とValue(値)の組み合わせの単純なデータ型で表現するデータベース。<br>
 目的は、単純なデータ問い合わせを高速化である。<br>
 大量データを高速に処理する必要のあるwebサービスで多く利用される。

 ### ドキュメント型データベース
 JSON(Javascript Object Notation)と呼ばれるJavaSCriptのオブジェクトを定義するためのデータ構造を利用する。<br>

```markdown
"member":{
  {"name":"山田　太郎",
  "prefecture":"東京都",
  },
  {
    "name":"koichi"
    "age":30
  }
}
```

## システム開発の工程と設計

システム開発は、よく建物や建造物をよく造る行為にたとえられている。<br>
開発は、建物と同様、いくつかの工程（ステップ）に分解することができる。<br>
1 要件定義　２　設計　３　開発　４　テスト<br>

### 要件定義
システムが満たすべき機能やサービスの水準、すなわち要件を決める工程。<br>
通常は、何らかの外部の顧客と要件を合意するが、パッケージソフトやWebサービスの場合は、自社内のみで要件を定義することもある。<br>

### 設計
定義された要件を満たすために必要なシステムを作るための設計(デザイン)を行う工程。建築でいえば設計図を書く作業に相当する。

### 開発
設計書にしたがってシステムを作る工程<br>
なお、システムを実際に作ることを、IT業界だと「実装する」という。<br>

### テスト
実装によって、組みあがったシステムが、本当に実用にたとえる品質であるかを試験する（テスト）する工程。<br>
目的に応じて、テストの種類とレベルにも何種類かあるが、大きくは機能的な品質に対するテストと、性能や信頼性といった非機能品質に対するテストの2種類に大別できる。<br>

上記四つの工程は、システム開発の最も基本的な分類であり、各工程はさらにいくつかのサブ工程に細分化される。<br>

## 設計工程と開発モデル
開発の進め方には（開発モデル）には、大きく二つ分かれる。<br>
一つがウォーターモデル、もう一つがアジャイルモデルである。<br>

### ウォーターフォールモデル
要件定義、設計、開発、テストといった各工程を、滝のように順次、一つずつ完了させていくシステム開発手法。<br>
滝が逆流することがないように、基本的に逆戻りすることがない。<br>
つまり、手戻りが難しいため、回収が高コストになりがち<br>
主に、大規模なシステム開発において採用されることが多いが、手戻りができないため、各工程の精密さが要求される。

### アジャイルモデル
最初に小さなプログラムを作り、顧客やユーザーに見せてフィードバックをもらい、それを取り入れてアップデートしていく。これらのサイクルを繰り返す循環的
な開発手法のことを指す。<br>
そのサイクルの期間は一般的に1か月以内とされており、アジャイル開発の開発るれーむワークである、スクラムでは、一つの開発サイクルのことをスクラムと呼ぶ。<br>
早い段階で、システムの具体的イメージを開発者や顧客と共有できるため、
要件の取りこぼしや、意思疎通の齟齬を防ぐことができる。途中で仕様が変更になっても対応しやすいなどのメリットがある。<br>
しかし、何度も同じ工程を繰り返す必要があるので、変更を繰り返すのち、発散してしまうなど収集がつかなくなるリスクがある。<br>
こうした特性を利用して、かつては小規模のシステムに適用していたが、現在は大規模開発をターゲットとした方法論も発展してきている。<br>
近年は、アジャイル開発をスムーズに実行するためCI/CDの自動化によってビルドやテストを高速に実行し、スムーズにデプロイすることを可能にする開発手法も発展してきている。<br>
※CI/CD・・・「コードを自動でテストして、自動で本番に反映する仕組み<br>
CI(Continuous Integration)・・・開発者が書いたコードのgithubにプッシュすると自動的にテストやビルドを行うこと。<br>
CD(Continuous Delivery)・・・CIでテストが通ったら、本番環境にリリースできるよう状態に自動的に持っていくこと。<br>
